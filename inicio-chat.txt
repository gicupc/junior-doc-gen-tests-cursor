================================================================================
 MANUAL OPERATIVO: ARCHITECT-BRAIN v4.6 (Cursor Edition)
 (Supply Chain Security + Testing 2026 BDD + Integration + AI-Assisted +
  Frontend + MCP + Database)
================================================================================

Sistema de scaffolding para Cursor que mantiene documentacion, codigo, tests,
decisiones, base de datos, frontend (accesibilidad + Core Web Vitals +
seguridad) y cadena de suministro (npm + GitHub Actions + OIDC publishing)
sincronizados a lo largo del ciclo de vida del proyecto.

v4.5 amplio el stack de testing con tres capas nuevas: BDD opt-in con Three
Amigos, integracion con Supertest/MSW/Testcontainers, y testing asistido por
IA con Playwright MCP/CLI + trazabilidad de prompts + gates GDPR / EU AI Act.

v4.6 anade defensa de cadena de suministro en 4 capas (dependency resolution
+ install-time execution + CI execution + publish path) tras el ataque a
TanStack del 11 mayo 2026, que demostro que el provenance SLSA valido NO es
garantia (84 versiones maliciosas en 42 paquetes con Build Level 3 valido).

Esta es la version Cursor del sistema. Existe tambien la version Windsurf:
https://github.com/gicupc/junior-doc-gen-tests

--------------------------------------------------------------------------------
 COMO SE USA EN 30 SEGUNDOS
--------------------------------------------------------------------------------
1. Copia toda esta estructura (.cursor/, docs/, prompts/, AGENTS.md,
   inicio-chat.txt, README.md) en la raiz del proyecto donde quieras trabajar.
2. Abre Cursor apuntando a la raiz del proyecto.
3. En el chat de Agent, escribe "/" y te aparecera el menu con los comandos.
4. Elige el comando segun la situacion (ver tabla debajo) y deja que la IA
   ejecute el protocolo.

La rule `architect-brain.mdc` se carga automaticamente con `alwaysApply: true`,
asi que las reglas globales del sistema estan presentes en cada chat sin que
tengas que invocarlas.

--------------------------------------------------------------------------------
 CONFIGURACION DE MCP SERVERS
--------------------------------------------------------------------------------
El archivo .cursor/mcp.json contiene plantillas COMENTADAS para los MCP servers
mas utiles en el flujo de desarrollo:

- postgres-template      -> BD Postgres (activado por defecto, renombrar y
                            configurar connection string)
- _figma-dev-mode        -> Figma Dev Mode MCP (descomentar para activar)
- _shadcn                -> shadcn MCP para acceso a bloques (descomentar)
- _chrome-devtools       -> Chrome DevTools MCP para auditoria runtime
- _playwright            -> Playwright MCP para validacion visual + E2E

Para activar un MCP:
1. Abre .cursor/mcp.json.
2. Renombra la clave que empieza por "_" (ej. "_figma-dev-mode" -> "figma-dev-mode").
3. Sustituye los placeholders (TU_FIGMA_PERSONAL_ACCESS_TOKEN, USER:PASS, etc.).
4. Reinicia Cursor para que detecte los cambios.
5. Documenta los MCP servers activos en docs/decisions-log.md como ADR.

REGLA CRITICA: En produccion, los MCPs de BD deben ir en modo READ-ONLY.
Para desarrollo local con datos de prueba, full-access es aceptable.

--------------------------------------------------------------------------------
 QUE HACE CADA COMANDO (detalle)
--------------------------------------------------------------------------------

/inicio
  Ejecuta On(project_start). Lanza la entrevista BMADT:
    B (Beneficio)  - objetivo y valor del proyecto.
    M (Mecanismo)  - stack tecnico e integraciones.
    A (Alcance)    - modulos del MVP.
    D (Datos)      - entidades y flujos criticos.
    T (Testing)    - nivel de cobertura: basico, estandar, exhaustivo.
  Al cerrar la entrevista, genera todos los docs en /docs/ (incluyendo
  decisions-log.md con las ADRs iniciales), ejecuta On(testing_setup),
  Y SI HAY BD, ejecuta tambien On(database_setup), Y SI HAY FRONTEND,
  ejecuta tambien On(frontend_setup): propone stack canonico 2026
  (Next.js + TS estricto + Tailwind v4 + shadcn + Biome + Zod), pregunta
  sobre flujo diseno-a-codigo (Figma con Dev Mode, capturas, sin diseno),
  recomienda MCP servers a activar (descomenta los bloques en .cursor/mcp.json),
  configura headers de seguridad, genera frontend-strategy.md. BLOQUEA la
  escritura de codigo de dominio hasta que apruebes.

/onboarding
  Codebase ajeno: ejercicio bootcamp, PR de companero, debug urgente.
  Mapa puntual en 6 secciones (stack, capas, traza de un endpoint,
  modelo de datos, tests, convenciones). NO genera docs ni adopta el
  proyecto. Diferencia explicita con /regularizar.

/regularizar
  Proyectos existentes sin documentacion. Escanea codigo + BD + frontend
  si existen, y ofrece:
    [1] Regularizar          - genera blueprints y docs sin tocar tests.
    [2] Evolucionar          - usa mejor /nueva directamente.
    [3] Reparar              - usa mejor /reparar directamente.
    [4] Regularizar + tests + auditar BD + auditar frontend - todo lo
                                anterior mas setup de tests, cobertura
                                de logica critica, auditoria completa
                                de BD si aplica, auditoria completa de
                                frontend si aplica.
  En [4]: tests-skill, legacy-testing-skill (si legacy detectado),
  database-skill (si BD), frontend-skill (si frontend), visual-testing-skill
  (si UI con interaccion). Genera audits y tickets correspondientes.

/hoy
  Ejecuta On(resume_project). Lee los docs clave (incluido
  database-strategy.md y frontend-strategy.md si existen), corre la suite
  de tests, detecta drift basico, y presenta un briefing de 10 lineas
  con sugerencia concreta para el dia.

/nueva
  Anadir funcionalidad. Activa interrupcion de seguridad que obliga a
  actualizar user-stories.md, roadmap.md, y si aplica blueprints.md,
  testing-strategy.md, database-strategy.md, frontend-strategy.md
  y decisions-log.md ANTES de tocar codigo. Al terminar, ejecuta
  obligatoriamente On(task_complete) con el checklist visible.

/reparar
  Corregir un bug. Si el bug no esta cubierto por tests, el protocolo
  obliga a escribir primero un test que lo reproduzca (rojo), y solo
  despues implementar el fix (verde). Para bugs en BD, ademas exige
  patron Expand-Contract si la reparacion es destructiva. Para bugs de
  a11y, exige primero un test con axe-core.

/decidir
  Revisar una decision pasada sin romper lo existente. Pregunta solo lo
  que cambia, calcula el impacto en los docs afectados (architecture,
  testing-strategy, database-strategy, frontend-strategy), pide
  confirmacion, anade nueva ADR al decisions-log, marca la anterior
  como superseded y propaga los cambios.

/sync
  Ejecuta On(sync_check). Escanea codigo + BD + frontend vs docs y
  reporta divergencias en 3 niveles (criticas, moderadas, menores).
  NO arregla automaticamente.

/cerrar
  Ejecuta On(task_complete) de forma explicita. Muestra el checklist
  de sincronizacion documental con cada doc marcado como actualizado
  o "sin cambios" con justificacion.

/cuestionar
  Audita la CALIDAD real de la suite de tests sin instalar herramientas.
  Filosofia: cobertura no es calidad. La IA propone 5+ mutaciones
  realistas, evalua si tu suite las detectaria, lista las que sobreviven
  y propone tests faltantes. Ofrece instalar mutation testing real
  (Infection, StrykerJS, PIT, mutmut) como decision opt-in con ADR.

/revisar-bd
  Audita la base de datos en 4 categorias:
    1. MODELO: tipos, indices, FKs, audit fields, normalizacion.
    2. SEGURIDAD: PII en plano, restricciones debiles, RLS, secrets.
    3. RENDIMIENTO: N+1, queries lentas, indices faltantes.
    4. CALIDAD DE DATOS: nulos, duplicados, rangos, integridad.
  Genera docs/database-audit.md y docs/investigacion_seguridad.md.
  Crea tickets por hallazgos criticos/altos. NO modifica BD ni codigo.

/revisar-frontend
  Audita el frontend en 4 categorias:
    1. STACK Y MANTENIBILIDAD: TS estricto, any, componentes gigantes,
       dependencias, mezcla de librerias UI, linter, tests.
    2. SEGURIDAD: secretos en cliente (NEXT_PUBLIC_*), dangerouslySet
       sin sanitizar, validacion solo en cliente, headers (CSP, HSTS),
       CORS, npm audit, codigo IA sin revision.
    3. RENDIMIENTO (Core Web Vitals): LCP, INP, CLS reales (CrUX) si
       hay despliegue; analisis estatico si no (imagenes, bundle,
       code-splitting, fonts).
    4. ACCESIBILIDAD WCAG 2.2 AA: alt, labels, contraste, teclado,
       landmarks, headings, focus visible, prefers-reduced-motion.
  Genera docs/frontend-audit.md y comparte hallazgos con
  docs/investigacion_seguridad.md. Crea tickets por hallazgos criticos
  y altos. Documenta TODOs de validacion manual con lector de pantalla.
  NO modifica codigo. La IA detecta, el humano decide, el humano repara.

/bdd   (NUEVO v4.5)
  Adopta Behavior-Driven Development en el proyecto con 10 pasos:
    1. Verificar aplicabilidad: stakeholders no tecnicos + disposicion
       a Three Amigos. Si NO se cumple, NO procede.
    2. Detectar stack y elegir herramienta segun regla 2026:
       - JS/TS con UI            -> playwright-bdd.
       - JS/TS sin UI (APIs)     -> @cucumber/cucumber (con scope).
       - JS/TS con Cypress legacy-> @badeball/cypress-cucumber-preprocessor.
       - .NET                    -> Reqnroll (SpecFlow discontinuado dic 2024).
       - JVM + APIs              -> Karate DSL.
    3. Instalar y configurar (carpetas features/ y features/steps/,
       scripts test:bdd* en package.json).
    4. Smoke test BDD que pase, luego borrarlo.
    5. Ofrecer primera sesion Three Amigos asistida con Example Mapping
       (Reglas - Ejemplos - Preguntas). Si >3 preguntas sin resolver,
       la historia NO esta lista para desarrollarse.
    6. Generar primer .feature siguiendo lenguaje del dominio (NO de la UI).
    7. VALIDACION HUMANA del .feature antes de generar step definitions.
    8. Generar step definitions reutilizando los existentes (no duplicar).
    9. Registrar ADR + tickets en roadmap.
    10. Cierre con On(task_complete).
  Filosofia: BDD sin Three Amigos es Gherkin disfrazado. Three Amigos sin
  BDD es una reunion mas. El valor aparece cuando las dos partes van juntas.

/revisar-supply-chain   (NUEVO v4.6)
  Audita la cadena de suministro de un proyecto JS/TS. Solo aplica si
  existe package.json. Las 4 capas de defensa:
    CAPA 1 - DEPENDENCY RESOLUTION:
      - Lockfile commiteado.
      - minimumReleaseAge activo (pnpm: 1440 min, npm: 2d, yarn: 3d, bun: 3d).
      - blockExoticSubdeps activo (pnpm 11).
      - Direct dependencies sin URLs exoticas (github:, tarball URLs).
      - trustPolicy: no-downgrade (pnpm).
    CAPA 2 - INSTALL-TIME EXECUTION:
      - strictDepBuilds activo (pnpm).
      - allowBuilds definido con paquetes legitimos.
      - ignore-scripts (npm clasico) si aplica.
      - Paquetes con lifecycle scripts revisados.
    CAPA 3 - CI EXECUTION (GitHub Actions):
      - NO pull_request_target con checkout del fork ("Pwn Request").
      - Cache no compartida entre PR y release.
      - Acciones pineadas a SHA (no tag mutable).
      - permissions: {} explicito.
      - Secrets no accesibles desde workflows de PR.
    CAPA 4 - PUBLISH PATH (OIDC):
      - OIDC en lugar de PAT.
      - Trusted publisher pineado a workflow + branch.
      - Branch protection en main con block force pushes.
      - 2FA activado en cuentas con permiso de publicacion.
      - Lockfile no expone _authToken ni _password.
  Cross-referencia el lockfile con bases publicas de paquetes
  comprometidos: TanStack mayo 2026 (GHSA-g7cv-rxg3-hmpx), Shai-Hulud
  (sep 2025 + ondas), axios 1.14.1/0.30.4, Trivy con tags repointed.
  Genera docs/supply-chain-audit.md con hallazgos por severidad. Crea
  tickets en roadmap.md por hallazgos criticos y altos. Comparte con
  docs/investigacion_seguridad.md. NO modifica .npmrc, pnpm-workspace,
  workflows ni dependencias. La IA detecta, el humano decide, el humano
  arregla.

  RESPUESTA A INCIDENTE - ORDEN CRITICO si detecta paquete comprometido:
    1. NO revocar token de GitHub todavia (vector rm -rf ~/ al revocar).
    2. Limpiar daemon de persistencia (gh-token-monitor en LaunchAgents/
       systemd/ Task Scheduler).
    3. Buscar payloads residuales (router_runtime.js, setup.mjs,
       tanstack_runner.js) en ~/.claude, ~/.vscode.
    4. Aislar la maquina de la red.
    5. Rotar TODAS las credenciales accesibles desde la maquina.
    6. Auditar versiones publicadas por la cuenta (worm self-propagation).
    7. Documentar como ADR de incidente.

--------------------------------------------------------------------------------
 REGLAS DE ORO
--------------------------------------------------------------------------------
1. CENTRALIZACION
   Toda documentacion (.md) vive exclusivamente en /docs/ de la raiz.

2. NO PROLIFERACION
   No se crean archivos nuevos para cambios individuales. Se editan los
   maestros: prd.md, user-stories.md, roadmap.md, blueprints.md,
   architecture.md, testing-strategy.md, database-strategy.md,
   frontend-strategy.md, decisions-log.md.

3. SINCRONIZACION
   Cada ticket finalizado debe marcarse con [x] en el roadmap, tener sus
   tests en verde, Y el checklist On(task_complete) visible en el chat.

4. TESTS COMO CONTRATO
   Un test en verde es un contrato verificable. Un test en .skip durante
   mas de una semana se convierte en ticket del roadmap.

5. DECISIONES TRAZABLES
   Si no esta en decisions-log.md, no existe como decision oficial.

6. ANTIDRIFT
   Ninguna tarea se cierra sin mostrar el checklist de sincronizacion.

7. LEGACY PRIMERO CONGELAR, LUEGO CAMBIAR
   En codigo legacy, nunca modificar sin haber capturado antes el
   comportamiento actual con characterization tests.

8. COBERTURA NO ES CALIDAD
   Un test que pasa no garantiza detectar bugs. La calidad real se
   audita con mutation thinking (/cuestionar).

9. BD PRIMERO AUDITAR, LUEGO CAMBIAR
   En proyectos con BD, ejecutar /revisar-bd antes de cambios destructivos.

10. FRONTEND PRIMERO MEDIR, LUEGO OPTIMIZAR
    En proyectos con frontend, ejecutar /revisar-frontend antes de
    optimizaciones a ojo. WCAG 2.2 AA y Core Web Vitals son criterios
    objetivos. Datos reales (CrUX, axe, lector de pantalla) > intuicion.

11. IMPLEMENTACION PASO A PASO
    Si un plan implica tocar mas de 2 archivos, aplicar
    On(implementation_phase) con paradas obligatorias entre archivos.

12. PROMPTS PROFESIONALES
    El usuario consulta @.cursor/skills/prompt-skill/SKILL.md para
    redactar prompts complejos siguiendo el patron RACEO.

13. CODIGO IA BAJO REVISION
    Outputs de v0/Lovable/Bolt/Figma Make pasan obligatoriamente por
    revision humana en Cursor/Windsurf/Claude Code antes de produccion.
    Tasa reportada: ~40-45% de vulnerabilidades en outputs sin revisar.

14. BDD SIN THREE AMIGOS NO SE ADOPTA (NUEVO v4.5)
    Gherkin sin conversacion previa con stakeholders es disfraz tecnico.
    Si el equipo no esta dispuesto a las conversaciones, mejor NO adoptar
    BDD. Registrar ADR "BDD descartado" y volver mas adelante.

15. TESTS IA TRAZABLES (NUEVO v4.5)
    Cada test generado por LLM tiene su *.prompt.md asociado, pasa code
    review humano y NUNCA se mergea por auto-aprobacion. Queries
    accesibles obligatorias (cero selectores fragiles).

16. AI LITERACY OBLIGATORIA (NUEVO v4.5)
    EU AI Act exige formacion en uso responsable de IA desde 2 febrero
    2025. Equipo formado antes de generalizar uso de agentes IA.

17. GDPR NO NEGOCIABLE (NUEVO v4.5)
    NO enviar PII a LLMs externos sin DPA valido. Para datos sensibles,
    usar LLMs locales (Ollama, vLLM) o proveedores con region europea
    y DPA firmado. Multas EU AI Act: hasta 35M€ o 7% facturacion global.

18. SUPPLY CHAIN EN 4 CAPAS (NUEVO v4.6)
    minimumReleaseAge (cooldown 1+ dia) + allowBuilds (deny by default) +
    GitHub Actions hardening (NO pull_request_target con checkout de fork,
    acciones pineadas a SHA) + OIDC trusted publishing pineado a
    workflow+branch. Provenance SLSA valido NO es garantia: el ataque
    TanStack del 11 mayo 2026 publico 84 versiones maliciosas con SLSA
    Build Level 3 provenance VALIDO desde el workflow legitimo. Defender
    en 4 capas, no en 1.

19. LIFECYCLE SCRIPTS DENEGADOS POR DEFECTO (NUEVO v4.6)
    allowBuilds (pnpm) o ignore-scripts (npm) reducen el vector clasico
    de worms npm: postinstall que descarga payload y roba credenciales
    (axios marzo 2026, Shai-Hulud, Trivy). Aprobaciones explicitas
    documentadas con justificacion (binario nativo legitimo) son lo unico
    que se permite ejecutar.

20. RESPUESTA A INCIDENTE CON ORDEN CRITICO (NUEVO v4.6)
    Si se detecta paquete comprometido instalado, NO revocar token de
    GitHub antes de limpiar daemon de persistencia. Algunos payloads
    (Mini Shai-Hulud, TanStack) ejecutan rm -rf ~/ cuando detectan
    revocacion. Orden correcto: limpiar persistencia -> aislar red ->
    rotar credenciales -> auditar blast radius -> documentar ADR.

 PROTOCOLO DE DEFENSA SI LA IA SE DESVIA
--------------------------------------------------------------------------------
Si en algun momento Cursor ignora el sistema (escribe codigo sin actualizar
docs, cierra una tarea sin checklist, modifica BD/frontend/supply-chain sin
auditar antes, adopta BDD sin Three Amigos, mergea tests IA sin *.prompt.md,
desactiva minimumReleaseAge sin ADR, etc.), usa este recordatorio:

  "Sigue el protocolo de .cursor/rules/architect-brain.mdc: consulta
   blueprints, testing-strategy, database-strategy, frontend-strategy,
   supply-chain-strategy y decisions-log, y cierra con checklist."

Si el problema persiste, invoca /cerrar para forzar el checklist del trabajo
reciente, /sync para auditar el estado real vs documentado, /revisar-bd o
/revisar-frontend para auditar BD o frontend si sospechas que se ha desviado,
/revisar-supply-chain si sospechas que las defensas de cadena de suministro
estan rotas o instalaste algo dudoso.
