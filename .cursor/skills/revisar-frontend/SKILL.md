# /revisar-frontend — Auditoria de frontend

Este comando NO modifica codigo. Examina el frontend (stack, seguridad, rendimiento, accesibilidad) para detectar problemas. Genera un reporte clasificado por severidad para que tu decidas que arreglar.

La filosofia: **la IA detecta. El humano decide. El humano repara.**

---

## Cuando usar este comando

- Heredas un proyecto con frontend y quieres saber su estado.
- Llevas tiempo desarrollando y sospechas problemas (bundle pesado, paginas lentas, accesibilidad rota).
- Antes de un release importante, como auditoria preventiva.
- Antes de pasar el frontend por una auditoria externa de accesibilidad (EAA, WCAG).
- Tras `/regularizar` opcion [4], se ejecuta automaticamente si hay frontend.
- Cuando el equipo ha usado vibe coding (v0/Lovable/Bolt) y necesita revision profesional antes de produccion.

---

## Requisitos previos

- El proyecto debe tener frontend (verificable en `package.json` con react/next/vue/svelte/angular/astro, o en la respuesta M del BMADT).
- Si tienes el sitio desplegado y accesible publicamente, la auditoria de Core Web Vitals sera con datos reales (CrUX). Si no, sera estatica.
- Si tienes Chrome DevTools MCP o Playwright MCP configurados en `.cursor/mcp.json`, la auditoria sera mas profunda.

---

## Pasos del protocolo

### Paso 1 — Determinar el alcance

Pregunta al usuario:

"Voy a auditar el frontend. Indica el alcance:
- [1] **Auditoria completa** — stack + seguridad + rendimiento + accesibilidad.
- [2] **Solo stack y mantenibilidad** — TypeScript, componentes, dependencias.
- [3] **Solo seguridad** — secretos en cliente, sanitizacion, validacion, headers.
- [4] **Solo rendimiento** — Core Web Vitals, bundle, imagenes, lazy loading.
- [5] **Solo accesibilidad** — WCAG 2.2 AA, axe-core, contraste, teclado.

Por defecto, [1] auditoria completa.

Indica tambien:
- URL publica del sitio (si la hay) para auditoria CWV real con PageSpeed Insights.
- Si quieres incluir testing manual con lector de pantalla en el reporte (queda como TODO documentado)."

### Paso 2 — Inventario inicial

Carga `@.cursor/skills/frontend-skill/SKILL.md` como guia.

Lee:
- `package.json` (dependencias, scripts).
- `tsconfig.json` (¿strict mode?, ¿noUncheckedIndexedAccess?).
- `next.config.js` / `vite.config.ts` / `astro.config.mjs` (headers, redirects).
- `components.json` si existe (shadcn).
- `biome.json` o `.eslintrc*` (¿reglas de a11y activas?).
- Estructura de carpetas (`app/`, `components/`, `pages/`).
- Variables de entorno publicas (`.env*`, busca `NEXT_PUBLIC_*`, `VITE_*`, `PUBLIC_*`).

Presenta un resumen breve antes de empezar la auditoria:

```
Stack detectado:
- Framework: [Next.js 15 / Astro 6 / SvelteKit / Vue+Nuxt / etc.]
- Lenguaje: [TypeScript estricto / TypeScript laxo / JavaScript]
- Estilado: [Tailwind v4 / CSS Modules / styled-components / etc.]
- Componentes: [shadcn / MUI / Chakra / propios]
- Linter: [Biome / ESLint+Prettier / ninguno]
- Tests: [Vitest+RTL / Jest+RTL / ninguno]
- Despliegue: [Vercel / Cloudflare / Netlify / ninguno detectable]
```

### Paso 3 — Auditoria 1: Stack y mantenibilidad

Detectar:

- **TypeScript no estricto o ausente** (CRITICO si proyecto enterprise; ALTA si proyecto personal).
- **`any` extendido**: contar ocurrencias. Si >5 sin justificacion documentada → ALTA.
- **Componentes gigantes**: archivos de componente >300 lineas → ALTA. Listar los top 5.
- **Mezcla de librerias UI**: shadcn + MUI + Chakra coexistiendo → ALTA (Frankenstein).
- **Dependencias obsoletas**: ejecutar `npm outdated`.
- **Linter ausente o sin reglas a11y**: ALTA.
- **Tests de componente ausentes**: si <30% cobertura → MEDIA.
- **Estado global mal usado** → BAJA.
- **Presencia de generadores IA sin revision**: detectar comentarios tipo `// generated by v0`, `// from lovable`, etc. → MEDIA-ALTA.

Output: tabla clasificada por severidad con archivos concretos.

### Paso 4 — Auditoria 2: Seguridad frontend

Detectar:

- **Secretos en cliente**: grep de `NEXT_PUBLIC_*`, `VITE_*`, `PUBLIC_*` con sufijos sospechosos. CRITICO.
- **`dangerouslySetInnerHTML` sin sanitizacion**: ALTA por cada ocurrencia.
- **Validacion solo en cliente**: CRITICO si los endpoints aceptan datos sin validar.
- **Headers de seguridad**: leer `next.config.js`/`vercel.json`/`netlify.toml`. Verificar CSP, HSTS, X-Frame-Options, Referrer-Policy. Cada uno ausente → ALTA. CSP ausente → CRITICO.
- **CORS abierto**: ALTA.
- **Dependencias vulnerables**: `npm audit --omit=dev --audit-level=high`.
- **API endpoints en cliente sin auth check**: ALTA.
- **Codigo IA sin revision**: marcar como zona de revision prioritaria.

Output: documento `docs/investigacion_seguridad.md` (compartido con auditoria de BD si existe).

### Paso 5 — Auditoria 3: Rendimiento (Core Web Vitals)

Si hay URL publica disponible:

- Lanzar **PageSpeed Insights** (o sugerir al usuario): `https://pagespeed.web.dev/analysis?url=URL`.
- Recoger LCP, INP, CLS de mobile y desktop, datos reales (CrUX) y de laboratorio.
- Reportar paginas que NO pasan los umbrales. Critico si LCP >4s, INP >500ms o CLS >0.25.

Si NO hay URL publica — analisis estatico:

- **Imagenes sin optimizar**: `<img>` plano en vez de `<Image>`.
- **Imagenes sin `alt`** y/o sin `loading="lazy"`.
- **Bundles potencialmente grandes**: `import _ from 'lodash'`, `import * as React`, `moment`.
- **Sin code-splitting**: rutas grandes sin `dynamic()` import.
- **Web fonts sin `font-display: swap`**.
- **Animaciones costosas**: `transform` en `top/left/width/height` en vez de `translate/scale`.
- **Tres.js / video / canvas sin carga diferida**.

Output: las 3-5 mejoras de mayor impacto, priorizadas. Para cada una mostrar archivo, codigo actual, codigo mejorado y mejora estimada.

### Paso 6 — Auditoria 4: Accesibilidad (WCAG 2.2 AA)

Si Playwright + axe esta configurado:

- Sugerir al usuario que ejecute la suite y pasar los resultados.
- Si la suite no existe, proponer crearla como ticket en el roadmap.

Si no hay tooling — analisis estatico del JSX/HTML:

- **Imagenes sin `alt`**: CRITICO.
- **`alt=""` en imagen informativa**: ALTA.
- **Botones sin texto accesible** (solo iconos sin `aria-label`): ALTA.
- **Inputs sin `<label>`**: CRITICO.
- **Errores de formulario no asociados** (sin `aria-describedby`): ALTA.
- **Contraste**: verificar tokens contra fondos. <4.5:1 → ALTA.
- **Headings desordenados**: paginas sin `h1` o saltos `h2 → h4`.
- **Landmarks ausentes**: ALTA.
- **Color como unica fuente de informacion**.
- **`prefers-reduced-motion` ignorado**: WCAG 2.3.3, ALTA.
- **Focus no visible**: `outline: none` sin alternativa: CRITICO.
- **`tabIndex` con valores positivos**: ALTA.

- Sugerir al usuario que ejecute manualmente: `npx @axe-core/cli https://tu-sitio.com` si hay despliegue.

Honestidad obligada: el tooling automatizado detecta ~30% de issues. Listar TODOs de validacion manual con NVDA/VoiceOver.

Output: tabla con findings, severidad WCAG (A/AA/AAA), criterio violado, archivo y propuesta de fix.

### Paso 7 — Generar el reporte ejecutivo

Crear `docs/frontend-audit.md`:

```
# Frontend Audit Report - [Fecha]

## Resumen ejecutivo
- Hallazgos criticos: X
- Hallazgos altos: Y
- Hallazgos medios: Z
- Hallazgos bajos: W

## Stack detectado
[Resumen del paso 2]

## Categoria 1 - Stack y mantenibilidad
[Tabla con findings]

## Categoria 2 - Seguridad
[Resumen + referencia a docs/investigacion_seguridad.md]

## Categoria 3 - Rendimiento (Core Web Vitals)
[Top 3-5 problemas con propuestas]
[Si hay datos CrUX: tabla LCP/INP/CLS por pagina]

## Categoria 4 - Accesibilidad (WCAG 2.2 AA)
[Tabla de findings con criterio WCAG violado]

## TODOs de validacion manual (no cubiertos por tooling)
- [ ] Test con lector de pantalla NVDA en pagina X.
- [ ] Test con lector de pantalla VoiceOver en pagina Y.
- [ ] Verificar contraste real en condiciones de luz baja.

## Recomendaciones priorizadas
1. [Hallazgo critico mas urgente] - ticket: ROADMAP-XXX
...
```

### Paso 8 — Crear tickets en el roadmap

Por cada hallazgo critico o alto: ticket nuevo en `roadmap.md` marcado como `[ ]`, linkeado al hallazgo.

NO arreglar nada automaticamente.

### Paso 9 — Sugerir siguiente paso

Pregunta al usuario:

"Auditoria completada. Hay X hallazgos criticos y Y altos.

¿Que prefieres ahora?
- [1] Empezar a reparar el hallazgo mas critico (uso `/reparar`).
- [2] Revisar primero el reporte completo (`docs/frontend-audit.md`).
- [3] Si la auditoria de a11y dio violaciones, instalar el tooling automatico (axe + Playwright) como ticket previo a las reparaciones.
- [4] Cerrar la auditoria sin acciones inmediatas (tickets quedan en el roadmap)."

### Paso 10 — Cierre

Ejecuta `On(task_complete)` con el checklist de sincronizacion documental.

---

## Reparacion de hallazgos

Cuando el usuario decida arreglar un hallazgo:

1. **NUNCA arreglar varios a la vez.**
2. Usar `/reparar` con el codigo del ticket.
3. Si la reparacion es de a11y, **escribir primero el test con axe** que la detecta antes del fix.
4. Si la reparacion es de seguridad, **considerar implicaciones en backend** (validacion solo en cliente nunca es suficiente).
5. Tras cada hallazgo arreglado, marcar el ticket como `[x]` en el roadmap.

---

## Que NO hace este comando

- **No modifica codigo** ni configuracion.
- **No instala dependencias** nuevas (las propone como tickets).
- **No despliega** ni toca infraestructura.
- **No genera tests** (eso lo hara `/reparar` o `/nueva` cuando se aborde el ticket).

Es un comando de **inteligencia, no de accion**.

---

## Regla de oro

> **Auditar antes de optimizar. Medir antes de cambiar. Validar a11y con humano antes de certificar.**

En frontend, optimizar a ojo es la causa mas comun de regresiones. Datos reales (CrUX, axe, lector de pantalla) > intuicion.
